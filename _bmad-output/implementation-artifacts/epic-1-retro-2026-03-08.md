## What Went Well

- Security posture improved through deny-by-default checks and stable GraphQL error code patterns.
- Team followed a regression-first pattern: implementation changes were paired with targeted backend and integration tests.
- Architecture boundaries were respected (frontend/backend/data/gameloop/integration), reducing accidental coupling.
- Review feedback was actioned quickly, including high-severity authorization and lifecycle correctness findings.
- Turn lifecycle observability and user-safe diagnostics became more actionable for host monitoring.

## Challenges and Friction Points

- Critical review findings surfaced late in Story 1.3 and Story 1.4, indicating earlier boundary checks should be part of definition-of-done.
- Integration reliability depended on runtime environment readiness (database availability) during validation.
- Several stories required follow-up hardening after initial completion, suggesting acceptance criteria interpretation can be tightened up front.

## Pattern Analysis Across Stories

### Recurring Positive Patterns

- Typed error semantics with stable `extensions.code` were consistently applied and expanded.
- Fail-closed authorization checks became a reusable implementation default.
- Regression tests were added near modified logic, improving confidence and change safety.

### Recurring Risk Patterns

- Host/member boundary rules and async lifecycle edge cases were easy to miss until review pressure.
- Integration tests can be brittle when environment assumptions are implicit.

### Breakthroughs

- Story 1.2 established a robust deny-and-audit access pattern that carried through subsequent stories.
- Story 1.4 introduced explicit turn-window contract handling and subscription diagnostics that improved operational clarity.
